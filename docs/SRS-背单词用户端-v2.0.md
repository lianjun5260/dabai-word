## 1. 引言

### 1.1 编写目的

本文档定义「大白陪你背单词」用户端的完整软件需求规格，作为产品、设计、开发、测试各角色的统一依据。**本版为全量设计，不存在V1裁剪**——所有功能必须完整实现。

### 1.2 适用范围

覆盖以下平台用户端：

- 微信小程序
- Android 原生应用
- iOS 原生应用
- 鸿蒙（HarmonyOS NEXT）原生应用

不包含管理后台（由独立后台 SRS 文档覆盖：`背单词系统-单词库管理后台_SRS_v2.50.docx`）。

### 1.3 术语与缩略语

| 术语 | 说明 |
|------|------|
| 学习组件 | 单词学习过程中的最小交互单元，如读单词、拼单词、英译汉等 |
| 新学 | 首次接触新单词的学习环节 |
| 复习 | 对已学单词的间隔重复复习环节 |
| 直选 | 从若干选项中直接选择正确答案的题型 |
| 全拼 | 通过字母键盘逐字母拼写出单词的题型 |
| 拼读 | 结合语音朗读与部分字母提示的拼写题型 |
| 录音评价 | 用户朗读单词后，系统通过语音识别给出发音评分 |
| 打卡 | 每日完成学习任务后的签到行为 |
| 词库 | 基于标签（word_tag）筛选出的单词集合，前端根据标签筛选单词主表得到 |
| 标签 | 学段/考试类别维度（如高考、四级、雅思），对应后台 word_tag 表 |
| 频次 | 单词在教材素材中出现的累计次数（total_frequency），支撑高频词优先推荐 |

## 2. 整体描述

### 2.1 产品视角

「大白陪你背单词」是一款面向 K12 学生及英语学习者的智能背单词应用。以 IP 角色"大白"为伴学形象，融合语音识别评分、9 种学习组件、间隔重复算法、频次驱动的智能推荐，提供趣味化、科学化的单词学习体验。

**核心价值主张**：

1. **多维学习**：9 种学习组件（读/拼/译/听）覆盖听、说、读、写全维度
2. **语音评测**：基于腾讯云 SOE 的 AI 录音评价，音素级纠错，帮助用户改善发音
3. **频次驱动推荐**：基于后台教材素材累计的单词频次（total_frequency），高频必考词优先学习
4. **伴学 IP**：大白形象贯穿全应用，学习过程不枯燥
5. **科学复习**：基于 SM-2 算法的间隔重复，最大化记忆效率
6. **全端覆盖**：一套代码（uni-app x / UTS）同时编译为小程序 + Android + iOS + 鸿蒙

### 2.2 用户特征

| 用户类型 | 特征描述 | 典型场景 |
|---------|---------|---------|
| K12 学生 | 按年级选词库，有明确学习进度 | 每日按计划新学+复习，打卡坚持 |
| 备考用户 | 有考试目标（四六级、考研、雅思托福等） | 针对性选择标签词库，强化训练 |
| 兴趣用户 | 无明确考试目标，以提升词汇量为目的 | 利用碎片时间学习 |

### 2.3 运行环境

| 平台 | 要求 |
|------|------|
| 微信小程序 | 基础库 >= 3.0.0，微信 >= 8.0 |
| Android | 最低 Android 8.0（API 26），目标 Android 14 |
| iOS | 最低 iOS 14.0，目标 iOS 17.0 |
| 鸿蒙 | HarmonyOS NEXT（API 12+） |

### 2.4 假设与依赖

- 用户设备有稳定网络（首次加载、同步、录音评测时）
- 词库数据由管理后台维护，客户端通过 API 获取
- 单词发音音频文件托管于 CDN
- 语音评测服务由腾讯云 SOE 提供
- 后台已实现单词生命周期管理（status 流转：审核中→待上架→已上架→已下架→黑名单）
- 前端查询可用单词时，SQL 条件为 `WHERE status=1 AND total_frequency > 0`

## 3. 技术架构决策

### 3.1 跨平台框架选型：uni-app x

**最终决策：uni-app x（Vue 3 + TypeScript + UTS）**

#### 选型依据

| 维度 | uni-app x | Taro | Flutter |
|------|-----------|------|---------|
| 小程序支持 | ✅ 原生支持，生态最成熟 | ✅ 原生支持 | ⚠️ 需转换层，体验差 |
| Android/iOS | ✅ UTS 编译原生，性能接近原生 | React Native 层 | ✅ 原生渲染 |
| **鸿蒙支持** | ✅ **4.61+ 正式支持 HarmonyOS NEXT，编译为 ArkTS** | ✅ 2025.6 开源支持 | ❌ 不支持 |
| 语音评测集成 | ✅ JS SDK + 原生插件 | ✅ JS SDK | ⚠️ 需原生桥接 |
| 学习成本 | 低（Vue 普及） | 低（React 普及） | 中（Dart 不普及） |
| 包体积 | 中 | 中 | 大 |
| 社区生态 | 国内最活跃 | 活跃 | 国际活跃 |
| **综合推荐** | **⭐⭐⭐** | ⭐⭐ | ⭐ |

**关键决策因素**：

1. **鸿蒙必选**：用户明确要求覆盖鸿蒙，uni-app x 是唯一成熟支持 HarmonyOS NEXT 的框架（4.61+ 版本编译为 ArkTS 原生代码）
2. **小程序为入口**：产品以小程序为主要获客渠道，uni-app x 的小程序生态最成熟
3. **Vue 生态**：国内开发者 Vue 技能储备丰富，降低团队学习成本
4. **腾讯云 SOE 兼容**：同为腾讯生态，小程序端 SDK 集成最顺畅

### 3.2 语音评测方案：腾讯云智聆口语评测 (SOE)

**最终决策：腾讯云 SOE（主）+ 讯飞 ISE（备用）**

#### 选型依据

| 维度 | 腾讯云 SOE | 讯飞 ISE |
|------|-----------|---------|
| 单词评测 | ✅ 音素级评测，含重音检测 | ✅ 音素级评测 |
| 句子评测 | ✅ 准确度/流利度/完整度/语调 | ✅ 多维度评测 |
| 小程序 SDK | ✅ **原生支持** | ⚠️ 无官方 SDK，需 WebSocket 适配 |
| Android/iOS SDK | ✅ 原生 SDK | ✅ 原生 SDK |
| 鸿蒙支持 | ⚠️ 待适配（可通过服务端中转） | ⚠️ 待适配 |
| 离线评测 | ✅ 提供离线 SDK（基础版） | ✅ 提供离线 SDK |
| 评测准确度 | 95%+ 与专家评分一致 | 95%+ 与专家评分一致 |
| 价格 | 按量计费，有免费额度 | 按量计费，有免费额度 |
| **综合推荐** | **⭐⭐⭐（小程序优先）** | ⭐⭐（App 端可用） |

**关键决策因素**：

1. **小程序原生 SDK**：SOE 是唯一提供微信小程序原生 SDK 的语音评测服务，无需复杂适配
2. **腾讯生态协同**：与微信登录、微信支付同属腾讯生态，集成顺畅
3. **离线能力**：提供离线 SDK，可满足离线学习场景
4. **鸿蒙适配策略**：鸿蒙端通过服务端中转（录音→上传→服务端调用 SOE→返回评分），无需客户端原生插件

**评测维度映射（对齐效果图）**：

| 评测场景 | SOE 模式 | 返回维度 | 客户端展示 |
|---------|---------|---------|-----------|
| 读单词 | 单词评测 | PronAccuracy/PronFluency | 0-100 分数 + 等级评语 |
| 读例句 | 句子评测 | PronAccuracy/PronFluency/PronCompletion | 0-100 分数 + 错误音素高亮 |
| 拼读模式 | 单词评测（带部分文本提示） | 同读单词 | 同读单词 |

### 4.1 用户系统

#### 4.1.1 登录

**FR-AUTH-001 微信一键登录 [P0]**

界面描述（参考效果图：01登录）：

- 页面顶部：应用 Logo + "大白陪你背单词" 标题
- 页面中部：大白 IP 形象插画
- 页面底部：绿色「微信一键登录」按钮（圆角，宽度撑满两侧留边距）
- 按钮下方：灰色辅助文字 "登录即同意《用户协议》和《隐私政策》"

交互流程：

1. 用户点击「微信一键登录」
2. 调起微信授权弹窗
3. 用户确认授权
4. 获取 openId 和基本信息（昵称、头像）
5. 首次登录自动创建账号 → 跳转选择年级页
6. 老用户直接进入首页

备选流程：

- 用户拒绝授权 → 提示「需要授权才能使用」，留在登录页
- 服务器异常 → 提示「登录失败，请重试」

#### 4.1.2 选择学段（标签）[2026-06-16 更新]

**FR-AUTH-003 选择学段 [P0]**

界面描述：

- 顶部标题："选择你的学习阶段" + 副标题"选择后将推荐适合你的词库"
- 学段标签以分组形式排列，3个分组：📘基础阶段 / 🎓大学考试 / 🌍留学考试
- 共 25 个标签，单选模式（点击即取消其他选中）：
  - 基础阶段（16个）：学前班、一年级~六年级、小升初、初一~初三、中考、高一~高三、高考
  - 大学考试（5个）：四级、六级、考研、专升本、专四专八
  - 留学考试（4个）：雅思、托福、GRE、GMAT
- 考试标签（小升初/中考/高考）使用虚线边框区分
- 底部固定按钮：「选择阶段，开始学习」（未选时灰色半透明，选中后变绿）

#### 4.1.4 单点登录与数据同步

**FR-USER-002 单点登录 [P0]**

同一账号同一时间仅允许在一台设备上登录。新设备登录后，旧设备自动踢下线。

**基本流程**：

1. 用户在设备 A 登录，正常使用
2. 用户在设备 B 使用相同账号登录
3. 服务端验证通过，生成新 token
4. 设备 A 的 token 立即失效
5. 设备 A 下一次 API 请求时返回 401，跳转登录页
6. 设备 A 页面提示："您的账号已在其他设备登录，如非本人操作请修改密码"

**被踢下线的处理**：

- 如果设备 A 正在学习中 → 弹窗提示"账号已在其他设备登录"，当前学习进度保存到本地，但不会同步到服务端（因为 token 已失效）
- 用户在设备 A 重新登录后 → 拉取服务端最新数据，本地未同步的学习记录以冲突提示方式让用户选择是否覆盖

**同步范围**（登录态下，同一设备内）：

- 学习进度（已学单词、掌握程度、SM-2 算法参数）
- 复习排期
- 收藏/生词本内容
- 打卡记录
- 个人设置

**同步策略**：增量同步，仅同步变更数据。

**FR-DICT-004 词库详情页 [P1]** [2026-06-17 新增]

页面标题：词库名称（点击可进入重命名模式），左上角 `<` 返回「我的词库」列表。

- **底部固定按钮栏**：
  - `📷 拍照`（品牌绿色） + `📁 上传文件`（蓝色）
  - 点击任一 → 弹出预处理进度弹窗 → 完成后素材加入列表 → 自动开始解析
- **删除词库**：底部居中浅灰色 `删除这个词库` 文字链接
  - 点击 → 弹出确认弹窗 → 显示影响（素材清空、单词数据丢失）→ 确认/取消
  - 确认后从 bankData 删除 → toast 提示 → 返回 P23
- ⚠️ 本页面不提供「开始学习」入口，学习入口仅在 P23 卡片上

**二、重命名**
- 点击标题 → 标题栏展开为 `<input>` + 保存按钮
- 保存时校验唯一性，重复提示「该词库名称已存在」
- 重命名同步更新 bankData 键名

**三、素材卡片设计（卡片自身即进度条）**

| 状态 | 背景 | 第一行右 | 进度指示 | 操作 |
|------|------|---------|----------|------|
| **解析中** | 绿色渐变（从左到右铺至当前进度%） | `暂停`（橙色）+ 百分比 | 卡片背景渐变 | 暂停、长按删除 |
| **已暂停** | 浅黄色（const）| `继续`（绿色）+ 百分比 | 同左，进度保留 | 继续、长按删除 |
| **已完成** | 白色（transparent）| `N词`（绿色文字）| 无 | 长按删除 |
| **失败** | 浅红色（const）| `重试`（蓝色）| 无 | 重试、长按删除 |
- **卡片高度**：所有状态高度一致（padding + 两行内容 + min-height 占位）
- **文件名显示**：去扩展名（如 `IMG_001.jpg` → `IMG_001`）
- **第二行**：相对上传时间（左，灰色小字）+ `·` 分隔 + 文件大小（如 `2.4MB`）
- **相对时间格式**：刚刚 → X分钟前 → X小时前 → 今天HH:mm → 昨天HH:mm → X天前 → MM-DD HH:mm → YYYY-MM-DD HH:mm（跨年）
- **文件大小**：模拟值，照片 1-5MB，文档 0.1-2MB

**四、素材交互**

| 操作 | 触发 | 行为 |
|------|------|------|
| **选中** | 点击素材卡片 | 单选模式，绿色左边框 + 浅绿底高亮；再点取消；解析中也可选；长按删除弹窗显示时不触发选中 |
| **暂停** | 点击右上角「暂停」 | 停止解析定时器，保留当前进度 |
| **继续** | 点击右上角「继续」 | 从暂停进度恢复解析 |
| **重试** | 点击右上角「重试」 | 进度归零重新解析 |
| **删除** | 长按素材 0.8s | 弹出自定义删除确认浮层（🗑 确认删除该素材？素材和已识别单词将清空），取消/删除 |
| 删除时解析中 | — | 自动 clearInterval 停止定时器，从 bankData 和 DOM 移除 |

**五、解析流程**
- 拍照/上传 → 预处理进度弹窗（居中覆盖层，文件图标 + 进度条 + 状态文字 + 取消按钮）
- 预处理完成 → 素材加入列表（status=parsing, progress=0）→ 自动启动 setInterval 模拟解析
- 解析进度每 350ms 更新一次（+8~15 随机增量）
- 解析完成 → status=completed → 随机 words（3-18个）→ 自动统计汇总到学习概览
- 空词库与有素材词库共用同一套 HTML 模板（DOM 结构完全相同，差异仅数据）

#### 4.8.2 会员中心

**FR-MINE-002 会员中心 [P0]**

**双重付费体系**：会员订阅 + AI 积分。

> **设计理念**：会员开通是基础订阅，覆盖核心学习功能；AI 积分用于按次计费的后台大模型调用（语音评测、AI 辅助等），因后台调用大模型有实际成本，需按量收费。

**权益对比表**：

| 功能 | 免费用户 | VIP 会员 |
|------|---------|---------|
| 每日新学单词数 | 最多 10 词 | 不限 |
| 学习组件 | 5 种基础组件 | 全部 9 种组件 |
| 语音评测 | 消耗 AI 积分 | 消耗 AI 积分（会员折扣价） |
| 词库数量 | 1 个标签 | 全部标签 |
| 学习统计 | 基础数据 | 高级统计+趋势图 |
| 离线学习 | 不支持 | 支持 |

**AI 积分体系**：

| 项目 | 说明 |
|------|------|

### 7.2 前端查询条件（对齐后台 SRS §3.3）

客户端查询可用单词（学习端）时，**必须**使用以下条件：

```
WHERE status = 1           -- 已上架
  AND total_frequency > 0  -- 有素材支撑（频次不为零）
```

此双重条件确保：
- 只有已上架且有教材支撑的单词才返回给学习者
- 频次归零的单词即使 status=1 也不可学习
- 后台管理员可自由上架/下架，不受频次影响

## 8. 学习组件完整规格

### LC-01 读单词（含录音评价）

**参考效果图：01_1~01_7**

**交互状态机（7 种状态）**：

| 状态 | 界面描述 | 触发 |
|------|---------|------|
| 初始态 | 单词+音标+发音按钮 + 底部「按住录音」按钮（灰色圆形麦克风图标） | 进入组件 |
| 录音态 | 按住按钮 → 红色脉冲动画 + "录音中" + 同时播放标准发音供跟读 | 用户按下录音 |
| 录音过短 | Toast 提示："录音时间太短，请再试一次" → 回到初始态 | 录音 < 3 秒松手 |
| 正在分析 | "正在分析..." loading 动画，等待 SOE 评测接口返回 | 录音完成 |
| 分析超时 | Toast 提示："分析超时，请重试" → 回到初始态 | 评测 > 10 秒 |
| 高分结果 | 分数（绿色大字号，如 90）+ "太棒了！" + 对勾 + 「继续」按钮 | 评分 >= 80 |
| 低分结果 | 分数（橙色/红色，如 55）+ "继续加油" + 标准音标提示 + 「再试一次」/「继续」按钮 | 评分 < 60 |

**业务规则**：

- 录音最短 3 秒
- 评测超时阈值 10 秒
- 评分范围 0-100
- 高分阈值 >= 80，低分阈值 < 60，60-79 为中等（显示"不错，还可以更好"）
- 离线模式跳过评测，直接标记为"已完成"
- SOE 调用模式：单词评测（WordEval）

## 9. SM-2 间隔重复算法规格

### 9.1 quality 评分映射

用户在学习组件中的表现映射为 SM-2 的 quality 值（0-5）：

| quality | 含义 | 映射条件 |
|---------|------|---------|
| 5 | 完美 | 直选/选择类一次答对 / 录音评测 >= 95 / 拼写一次正确 |
| 4 | 容易 | 直选/选择类一次答对但犹豫时间 > 5s / 录音评测 80-94 / 拼写第二次正确 |
| 3 | 正常但吃力 | 多次尝试后答对 / 录音评测 60-79 / 点击"认识"但不确定 |
| 2 | 困难 | 答错后记住答案 / 录音评测 40-59 / 点击"不认识" |
| 1 | 很困难 | 连续 2 次答错 / 录音评测 < 40 |
| 0 | 完全不记得 | 新词翻转后直接点"不认识" / 全拼连续 2 次完全错误 |

### 9.2 核心算法逻辑

```
输入：当前 easeFactor, interval, repetitions, quality

如果 quality >= 3（回忆成功）：
  如果 repetitions == 0：interval = 1（天）
  如果 repetitions == 1：interval = 6（天）
  如果 repetitions >= 2：interval = interval × easeFactor
  repetitions += 1
  easeFactor = max(1.3, easeFactor + 0.1 - (5-quality)×(0.08+(5-quality)×0.02))

如果 quality < 3（回忆失败）：
  repetitions = 0
  interval = 1
  easeFactor 不变（保持当前值，不下调）

输出：nextReviewAt = now + interval 天
      更新后的 easeFactor, interval, repetitions
```

### 9.3 用户可感知行为

- quality >= 3：复习间隔逐步拉长（1天 → 6天 → 更长）
- quality < 3：间隔重置为 1 天，第二天再复习
- easeFactor 最低 1.3，即最长间隔不超过当前间隔的 1.3 倍增长
- 高频词（total_frequency 排名 TOP 30%）在首次新学时获得额外 1 次复习

## 10. 学习模式与组件调度策略

### 10.1 设计理念

学习模式由**后台配置表**驱动，支持动态调整。客户端不硬编码组件编排逻辑，而是从后台拉取当前模式配置后执行。

**四种学习模式**：

| 模式 | 编号 | 说明 | 组件编排 |
|------|------|------|---------|
| 简单 | MODE-EASY | 最少组件，适合快速过词 | 固定配置，写死组件列表 |
| 标准 | MODE-STANDARD | 常规学习，覆盖核心维度 | 固定配置，写死组件列表 |
| 困难 | MODE-HARD | 全组件强化，适合深度学习 | 固定配置，写死组件列表 |
| 智能 | MODE-SMART | 算法动态调度（默认模式） | 算法决定，初期随机调度 |

> **用户可选**：在"设置学习目标"页面切换学习模式，默认"智能"模式。

### 10.2 后台配置表设计

**learning_mode_config（学习模式配置表）**：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | int | 主键 |
| mode_code | varchar(20) | 模式编码：EASY / STANDARD / HARD / SMART |
| mode_name | varchar(20) | 模式名称：简单 / 标准 / 困难 / 智能 |
| description | varchar(200) | 模式描述 |
| new_learn_components | json | 新学阶段组件编排列表 |
| review_components | json | 复习阶段组件编排列表 |
| is_active | tinyint | 是否启用（1=启用，0=停用） |
| sort_order | int | 排序权重 |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |

### 10.3 各模式组件编排规则

#### 10.3.1 简单模式（MODE-EASY）

**定位**：最少交互，快速浏览+基础巩固，适合时间紧张或只想快速过词的用户。

**新学阶段编排**：

```
看词义（卡片翻转） → LC-04 英译汉（选择题） → LC-05 汉译英-直选 → 下一个词
```

**复习阶段编排**：

```
LC-04 英译汉（选择题） → LC-05 汉译英-直选 → 下一个词
```

**涉及组件**：LC-04、LC-05（2 种），不含拼写和语音

---

#### 10.3.2 标准模式（MODE-STANDARD）

**定位**：覆盖听/说/读/写四维，适合大多数用户。

**新学阶段编排**：

```
看词义（卡片翻转） → LC-01 读单词 → LC-04 英译汉 → LC-05 汉译英-直选 → LC-02 拼单词 → 下一个词
```

**复习阶段编排**：

```
LC-08 听力(词) → LC-04 英译汉 → LC-05 汉译英-直选 → LC-02 拼单词 → 下一个词
```

**涉及组件**：LC-01、LC-02、LC-04、LC-05、LC-08（5 种）

---

#### 10.3.3 困难模式（MODE-HARD）

**定位**：全组件强化训练，每种组件都会遇到，适合备考冲刺。

**新学阶段编排**：

```
看词义（卡片翻转） → LC-01 读单词 → LC-03 读例句 → LC-02 拼单词
→ LC-04 英译汉 → LC-05 汉译英-直选 → LC-06 汉译英-全拼
→ LC-07 汉译英-拼读 → 下一个词
```

**复习阶段编排**：

```
LC-08 听力(词) → LC-09 听力(句) → LC-04 英译汉 → LC-05 汉译英-直选
→ LC-06 汉译英-全拼 → LC-07 汉译英-拼读 → LC-01 读单词 → 下一个词
```

**涉及组件**：LC-01 ~ LC-09（全部 9 种）

---

#### 10.3.4 智能模式（MODE-SMART，默认）

**定位**：算法动态调度，根据用户对每个词的掌握程度和弱项，智能匹配最需要的学习组件。

**当前策略（V1 阶段 — 随机调度）**：

由于缺乏足够的用户学习数据来训练智能算法，**初期采用随机调度策略**：

```
新学阶段：看词义 → 从 LC-01~LC-07 中随机选择 3~4 个组件（保证覆盖不同维度）
复习阶段：从 LC-01~LC-09 中随机选择 2~3 个组件（优先选择历史正确率低的维度）
```

**随机调度规则**：

1. 每个词至少覆盖 2 个不同维度（听/说/读/写中至少 2 个）
2. 连续 2 个词不使用完全相同的组件组合（避免单调感）
3. 录音评测组件（LC-01/LC-03/LC-07）不连续出现（防止用户疲劳）
4. 新学阶段前 3 个词固定使用标准模式编排（给用户稳定的初始体验）

**智能调度策略（V2 阶段 — 数据驱动，待实现）**：

当积累足够学习数据后，切换为基于数据的智能调度：

```
输入：用户对当前词的历史表现（各组件正确率）、掌握度（SM-2 easeFactor）、
      整体弱项维度（听/说/读/写哪个维度正确率最低）

调度逻辑：
1. 计算用户在听/说/读/写四个维度的历史正确率
2. 找到最弱维度 → 优先调度该维度的组件
3. 对当前词的掌握度越低 → 安排更多组件（最多 5 个）
4. 对当前词的掌握度越高 → 安排更少组件（最少 2 个），跳过已熟练维度
5. 录音评测（消耗 AI 积分）仅在"说"维度正确率 < 80% 时调度
```

> **注**：智能模式 V2 的具体算法参数需要根据上线后的学习数据统计分析后确定，不在当前 SRS 中定义。但后台配置表和接口已预留扩展能力。

### 10.4 学习组件业务规则（待逐项确认）

以下为学习组件的通用业务规则框架。**具体交互细节（答对/答错跳转、连续错误降级、中断恢复等）需要通过基础交互原型逐项确认**，确认后补充到本文档。

| 答错后行为 | 同词重试 / 标记后继续？显示正确答案时长？ | 待确认 |

## 11. 学习单元系统（v3.0 新增）

### 11.1 概念

**学习单元（Learning Unit）**：一个单词在一个学习组件中的一次练习。例如"abandon 在 LC-01 读单词"是一个学习单元。

**学习任务（Learning Task）**：一次完整的学习会话，由 N 个学习单元组成。N = 单词数 × 每词组件数。

**学习明细**：学习任务中所有学习单元的线性列表。

### 11.2 创建规则

| 规则 | 说明 |
|------|------|
| 触发时机 | 用户在 P07 今日学习预览点击「开始学习」 |
| 每日次数 | 每个任务仅创建一次学习明细（todayTask.created=false→创建并播动画→标记true） |
| 任务完成 | 全部学习单元状态为 done → 任务结束 |
| 再次进入 | 任务未完成时再次点击「开始学习」→ 直接跳到中断位置，不播动画 |
| 新一轮 | 任务完成后再次点击 → todayTask.created=false，重新创建并播动画 |
| 次日重置 | 隔天重新计算 todayTask，created 重置为 false |

### 11.3 学习模式→组件映射

### 12.1 通用答题判定规则（v3.6 新增）

**适用范围**：全部9个学习组件，后续新增组件也必须遵守。

**后台判定（发给服务端的结果）**：

| 用户行为 | 前端status | 后台记为什么 |
|------|:--:|:--:|
| 选了选项 → 正确 | `correct` | ✅ 正确 |
| 选了选项 → 错误 | `wrong` | ❌ 错误 |
| 没选，直接点「查看答案」 | `showAnswer` | ❌ 错误（视为未掌握） |
| 拼写正确（满字自动判） | `correct` | ✅ 正确 |
| 拼写错误（满字自动判） | `wrong` | ❌ 错误 |
| 未输完点「单词详情」 | `wrong` | ❌ 错误 |

> **原则**：只要不是 `correct`，后台一律记为错误。

**前端展示（给用户看的）**：

| 场景 | 前端效果 |
|------|------|
| 选对 / 拼对 | 绿色 ✓ + 例句/答案揭示 |
| 选错 | 红色 ✗ 标错项 + 绿色 ✓ 标正确项 + 例句/答案揭示 |
| 直选组件点「查看答案」(showAnswer) | **只揭示正确答案（绿色 ✓）**，不标红色 ✗，不惩罚式显示 |
| LC-06 未输完点详情 | 红色 ✗ + 已输入字母标红 + 答案框揭示 |

> **核心区别**：`showAnswer` 和 `wrong` 后台都算错误，但前端展示不同——用户主动放弃不应受红叉惩罚。

### 13.6 干扰词生成

1. 从 wdDB 中优先选取同词性的单词
2. 排除当前正确答案
3. 按长度相近（±3字符）排序
4. 随机取前3个
5. 若不足3个，放宽词性限制
6. 若仍不够，返回 toast「词库数据不足」

### 19.4 词块数据结构

单词按自然拼读法拆分为音节/拼读块，每个块有主文本（拼写）和副文本（音标提示）：

```javascript
// 示例：industrious → [in, dus, tri, ous]
var lc07Data = {
  'industrious': {
    chunks: ['in','dus','tri','ous'],         // 正确词块
    chunkSubs: ['ɪn','dʌs','tri','əs'],       // 音标提示
    distractors: [                              // 干扰块
      {text:'ty',sub:'ti'}, {text:'ti',sub:'ti'},
      {text:'ous',sub:'es'}, {text:'di',sub:'di'},
      {text:'dual',sub:'d3uel'}, {text:'try',sub:'tri'}
    ]
  }
};
```

**选项池 = 正确块 + 干扰块**，随机打乱排列。

**干扰项设计原则**：
- 形近/音近拼读块（如 `ous/es` vs 正确 `ous/tri`）
- 同根词变形片段（如 `try/tri` 与正确 `tri`）
- 干扰数量约6~8个，总选项池约10个

### 19.5 词块下划线行渲染

- `renderChunkLine()` 函数
- 生成 N 个 `<div class="chunk-cell">` 格子（N=正确词块数量）
- idle/selecting：已选词块=黑色显示，未选=透明（只显示下划线）
- correct：词块绿色（class=chunk-cell correct-chunk）
- wrong：正确位置词块=绿色，错误位置词块=**红色**（class=chunk-cell wrong）

### 19.6 词块键盘状态

| 状态 | CSS类名 | 外观 |
|------|---------|------|
| 可选 | `chunk-key` | 白底灰边 |
| 已选中 | `chunk-key selected` | 绿底绿边绿字 |
| 答后-正确块 | `chunk-key result-correct disabled` | 绿底绿边+不可点击 |
| 答后-选中干扰块 | `chunk-key result-wrong disabled` | 红底红边+不可点击 |
| 答后-未选中干扰块 | `chunk-key disabled` | 灰底+不可点击 |
| showAnswer-正确块 | `chunk-key result-correct disabled` | 绿底绿边+不可点击 |
| showAnswer-其余 | `chunk-key disabled` | 灰底+不可点击（不标红，不惩罚） |

### 19.7 判定算法

```
for (i = 0; i < chunks.length; i++) {
  optIdx = userChunks[i]
  opt = options[optIdx]
  if (opt.isCorrect && opt.correctIndex === i) → 该位置正确
  else → 该位置错误
}
全部正确 → ✓ 绿色
有错误 → ✗ 红色
```

**关键**：不仅要选对了正确的词块文本，还要**选对了顺序位置**（`correctIndex === i`）。

  example: { en:'', cn:'' },
  chunks: [],              // 正确词块数组 (如 ['in','dus','tri','ous'])

};
